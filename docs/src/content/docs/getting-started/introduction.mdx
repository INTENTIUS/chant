title: Introduction
description: What chant is and why it exists
---

chant is a TypeScript-first infrastructure tool. You define cloud resources as plain TypeScript objects, chant type-checks them against a provider lexicon, and emits deployment-ready output. No YAML, no string templating, no deploy step, no state to manage.

```
TypeScript source → chant build → provider-specific output
```

The product is that synthesis core. Everything past it — state, apply, drift, reconcile — is an opt-in bolt-on: **pluggable lexicons in, pluggable lifecycles out, pure synthesis in the middle.** Bring your own lifecycle, and chant makes it durable. See [Lifecycle Models](/chant/concepts/lifecycle-models/) for the full picture.

"No state to manage" means no authoritative state file to host, lock, or trust — not the absence of a change set. chant computes a precise create/update/delete plan against the live system using ownership markers that live on the cloud resource, so you get the plan without the file ([State and Governance](/chant/concepts/governance/)). And when an apply needs to be durable — approval gates, rollback, crash-resume — chant compiles your orchestration to [durable workflows](/chant/concepts/durable-workflows/): Temporal-native when you want durability, zero-dependency when you don't.

You get real `import`/`export`, editor autocompletion, and semantic lint rules that catch misconfigurations before deployment — all in a single local pass.

## Choose Your Path

- **Synthesis only** — You have CI, ArgoCD, Terraform, or another deployment tool. chant produces typed, linted artifacts and hands off. [Quick Start](/chant/getting-started/quick-start/) covers everything. Lexicons are available for [AWS](/lexicons/aws/), [Azure](/lexicons/azure/), [GCP](/lexicons/gcp/), [Kubernetes](/lexicons/k8s/), [Helm](/lexicons/helm/), [Docker](/lexicons/docker/), [GitHub Actions](/lexicons/github/), [GitLab CI](/lexicons/gitlab/), and [Temporal](/lexicons/temporal/).
- **Synthesis + deployment** — You want durable deployments with phase visibility, gate steps, and rollback. Declare phases in `*.op.ts`, `chant build` generates Temporal worker code, `chant run` starts the workflow. Start with [Quick Start](/chant/getting-started/quick-start/), then the [Ops guide](/chant/guide/ops/). Not sure if Ops fits? See [Choosing Your Deployment Model](/chant/concepts/deployment-paths/).
- **Create a lexicon** — [Philosophy](/chant/concepts/philosophy/) for the design rationale, then [Lexicon Authoring](/chant/lexicon-authoring/overview/) to build a plugin.
- **Contribute to chant** — [Development Setup](/chant/contributing/development/), then the [Architecture Overview](/chant/architecture/overview/).
